## Executive Summary

The site has excellent content quality in its deeper pages (romanization guide, IPA guide, batchim, Dutch guide) but is held back by a cluster of technical issues that compound each other: a duplicate canonical injected from the root layout, a sitewide hreflang override pointing every page to `/en/consonants`, no security headers, no brotli compression, and a 1.28 MB uncompressed hero image. These are all fixable in the codebase with precision edits.

On the content side, the two highest-traffic pages (`/en/consonants` and `/en/vowels`) are nearly all interactive component with no indexable prose, making them uncitable for the site's primary target queries. The Dutch locale infrastructure exists in the codebase but is effectively invisible to search engines.

### Critical Issues

**Duplicate canonical tags on every locale page**
The root layout (`__root.tsx`) injects a `createSeoHead()` call with a canonical pointing to `https://korean.youcanlearn.it/` (homepage), and the `$locale/$slug.tsx` route injects a second `createSeoHead()` with the correct page-specific canonical. When Google encounters two conflicting canonicals, it may ignore both and collapse the entire site to the homepage URL.

```
<!-- Root layout (WRONG for child routes) -->
<link rel="canonical" href="https://korean.youcanlearn.it/"/>

<!-- Child route (correct) -->
<link rel="canonical" href="https://korean.youcanlearn.it/en/consonants"/>
```

**Source:** `src/routes/__root.tsx` lines 39–61

---

**Sitewide hreflang override pointing to consonants**
The root layout unconditionally outputs `hreflang="en"` → `/en/consonants` and `hreflang="nl"` → `/nl/consonants` on every page — including the homepage, quiz, grammar, and vocabulary pages — before each child route's own hreflang tags. Google sees `/en/consonants` as the language alternate for the entire site.

**Source:** `src/routes/__root.tsx` — three static `<link rel="alternate">` lines in the root `head()`

### High Issues

**`?from=legacy` query parameter on redirect destinations**
`/consonants` → 301 → `/en/consonants?from=legacy`. The server returns 200 with the correct canonical, but the parameter creates a parallel indexable URL and is passed unnecessarily to clean internal navigations.

**No security headers**
Every standard security header is absent from HTTP responses. The server returns only `alt-svc`, `content-type`, and `date`.

| Header | Risk |
|---|---|
| `Strict-Transport-Security` | High — no forced HTTPS upgrade signal |
| `Content-Security-Policy` | High — XSS attack surface |
| `X-Frame-Options` | Medium — clickjacking risk |
| `X-Content-Type-Options` | Medium — MIME sniffing |
| `Referrer-Policy` | Low — referrer data leaks |

**Homepage internal links use non-locale paths**
`<a href="/consonants">` links on the homepage trigger 301 redirect chains to `?from=legacy` destinations. Direct links to `/en/consonants` would skip the redirect entirely.

**Source:** `src/routes/index.tsx`

### Medium Issues

- Sitemap hreflang alternates only present on consonants and vowels; all other EN/NL pairs (blocks, batchim, grammar, romanization-guide, ipa-guide, dutch-guide) are missing `<xhtml:link>` entries
- No `<lastmod>` dates on any sitemap entry
- 404 page does not emit `<meta name="robots" content="noindex">`

### Low Issues

- IndexNow not implemented
- Trailing-slash redirect uses 307 instead of 301
- `PronunciationModel` chunk unnecessarily preloaded on homepage
- Mixed trailing slash usage in sitemap (homepage has `/`, all others omit it)

## Schema / Structured Data — 40 / 100

### Existing Schema

One JSON-LD block sitewide (all pages, injected from `__root.tsx`):

```json
{
  "@context": "https://schema.org",
  "@type": "WebSite",
  "name": "한글 배우기",
  "alternateName": "Learn Hangul",
  "url": "https://korean.youcanlearn.it",
  "inLanguage": ["en", "nl"],
  "about": { "@type": "Thing", "name": "Hangul", "alternateName": "Korean alphabet" },
  "educationalLevel": "Beginner"
}
```

### Validation Issues

| Property | Issue |
|---|---|
| `educationalLevel` on `WebSite` | Invalid — this property belongs on `LearningResource` or `Course`, not `WebSite`. Google silently ignores it. |
| `about` typed as `Thing` | Valid but imprecise — should be `@type: Language` |
| `SearchAction` | Missing — would enable Google Sitelinks Search Box |
| `publisher` | Missing — recommended for editorial trust signals |

### Missing Schema Opportunities

**LearningResource (per content page)** — High priority
All content pages (consonants, vowels, blocks, batchim, romanization-guide, ipa-guide, grammar) should have page-level `LearningResource` schema. This is eligible for Google rich results.

```json
{
  "@context": "https://schema.org",
  "@type": "LearningResource",
  "name": "Consonants — Korean Hangul",
  "url": "https://korean.youcanlearn.it/en/consonants",
  "inLanguage": "en",
  "educationalLevel": "Beginner",
  "learningResourceType": "Interactive Resource",
  "teaches": "Korean Hangul consonants",
  "isAccessibleForFree": true,
  "about": { "@type": "Language", "name": "Korean", "alternateName": "한국어" },
  "publisher": { "@type": "Organization", "name": "한글 배우기", "url": "https://korean.youcanlearn.it" }
}
```

**FAQPage (romanization guide)** — High priority
The `/en/romanization-guide` page has question-structured H2 headings that map exactly to PAA queries with high AI Overview rates. FAQPage JSON-LD here would directly generate People Also Ask entries and AI Overview snippets.

**Quiz as LearningResource** — Medium priority
```json
{
  "@type": "LearningResource",
  "learningResourceType": "Quiz",
  "teaches": "Korean Hangul recognition and recall"
}
```

**Not recommended:**
- HowTo schema — deprecated from Google rich results September 2023
- FAQPage for Google rich results on non-government/healthcare sites (August 2023 restriction)

## Performance / Core Web Vitals — 45 / 100

### Critical Issues

**No brotli/gzip compression**
The main JS bundle is 426 KB served **raw**. With brotli it compresses to ~110 KB (74% reduction). The server is not serving any compression even when `Accept-Encoding: br` is sent.

| Asset | Served Size | Compressed Estimate |
|---|---|---|
| `index-CmC5MZ3u.js` | 426 KB | ~110 KB brotli |
| `index-CRM8P_X8.css` | 35 KB | ~8 KB brotli |

**Hero image is 1.28 MB PNG with no modern format alternative**

| Image | Size | WebP Estimate |
|---|---|---|
| `/social/hangul-og-syllables.png` | 1.28 MB | ~140 KB |
| `/social/hangul-og.png` | 1.31 MB | ~150 KB |

`fetchpriority="high"` and `<link rel="preload">` are correctly set — the LCP mechanics are right, the image size is the problem.

### LCP Risk: HIGH

LCP candidate is the hero image. Primary risks:
1. 1.28 MB PNG with no WebP/AVIF fallback
2. No `Cache-Control` header on image assets (ETag only — must revalidate every visit)
3. No CDN — bare Hetzner VPS in Germany adds 100–300ms latency for non-EU users

### CLS Risk: MEDIUM

- Noto Sans KR and Noto Serif KR loaded via Google Fonts with `display=swap` — FOUT at 14rem hero causes measurable reflow
- No `size-adjust` / `ascent-override` fallback font descriptors
- `preconnect` hints placed at positions 31–32 in `<head>`, after the font stylesheet at position 4 — hints are effectively wasted

### INP Risk: MEDIUM

- No compression means 426 KB JS must parse before interactions become live
- `LegacyRefiner` `useEffect` navigation fires on mount for legacy-redirect arrivals — potential hydration delay
- Spaced repetition weight recalculation on quiz answer unconfirmed for synchronous blocking

### Caching

| Asset Type | Cache-Control | Assessment |
|---|---|---|
| JS assets (`/assets/*.js`) | `immutable, max-age=31536000` | Correct |
| CSS assets (`/assets/*.css`) | `immutable, max-age=31536000` | Correct |
| Images (`/social/*.png`) | None (ETag only) | Missing |
| HTML pages | None | Expected for SSR |
